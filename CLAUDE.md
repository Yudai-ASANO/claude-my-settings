# グローバル設定

## 基本方針
- Conventional Commits 形式でコミットする（feat: / fix: / refactor: / test: / docs: / chore:）
- 長いセッションでは `/clear` を活用してコンテキストをリフレッシュする

## Codex Plugin 運用

> 以下のルールは `codex-plugin-cc` がインストール・有効化済みの場合にのみ適用する。
> 単純なタスク（1ファイル・数行の修正、設定変更、ドキュメント編集等）は Claude が直接実行してよい。

### アーキテクチャ

```
ユーザー → Claude Code（司令塔・設計・統合）
  ↓
  /codex:review, /codex:rescue → Codex Plugin（レビュー・実装・調査）
  ↓
  Claude Code（検証・統合・コミット判断）
```

### コマンドパターン

| 目的 | コマンド | 説明 |
|------|---------|------|
| コードレビュー | `/codex:review` | 未コミット変更またはブランチ差分のレビュー |
| 設計批評レビュー | `/codex:adversarial-review` | トレードオフ・障害モードへの挑戦 |
| タスク委譲 | `/codex:rescue` | バグ調査、実装、リサーチの委譲 |
| ジョブ状態確認 | `/codex:status` | 実行中/完了済みジョブの一覧 |
| 結果取得 | `/codex:result` | 完了ジョブの出力取得 |
| ジョブキャンセル | `/codex:cancel` | 実行中ジョブのキャンセル |
| セットアップ確認 | `/codex:setup` | インストール・認証状態の確認 |

### 委譲の判断基準

#### Codex Plugin に委譲する条件（以下のいずれかを満たす場合）
- コードレビュー → `/codex:review` または `/codex:adversarial-review`
- 新規コード生成が20行以上 → `/codex:rescue`
- 複数ファイルにまたがる変更 → `/codex:rescue`
- バグの根本原因分析 → `/codex:rescue`
- セキュリティ脆弱性スキャン → `/codex:adversarial-review`

#### Claude が直接実行する条件
- 1ファイル・数行の修正
- 設定ファイルの編集（YAML, JSON, TOML 等）
- ドキュメント・README の作成・更新
- git 操作（コミット、ブランチ、マージ）
- ユーザーへの説明・質疑応答

### 委譲ルール

#### 1. レビューは Codex Plugin に任せる

コード変更後のレビューは `/codex:review` で実施する。Claude はレビュー結果を受けて最終判断する。
設計判断の妥当性を確認したい場合は `/codex:adversarial-review` を使用する。

#### 2. バックグラウンドジョブの活用

`/codex:rescue` で委譲したタスクはバックグラウンドで実行される。
`/codex:status` で進捗確認、`/codex:result` で結果取得する。

#### 3. Claude はパススルーしない

Codex の出力をそのままユーザーに見せない。必ず検証・統合した上で報告する。

### レビューゲート

`/codex:setup --enable-review-gate` で有効化すると、Claude の Stop 時に自動的に
Codex レビューが実行される。問題が見つかった場合はセッションがブロックされる。

### セキュリティに関する注意

- API キーや秘密情報を Codex へのプロンプトに含めない
- Codex が生成したコードのセキュリティレビューは必ず実施する

## Gemini CLI サブエージェント運用

> 以下のルールは Google Gemini CLI がインストール・認証済みの場合にのみ適用する。
> Gemini CLI は**外部情報の取得**専用 — Web 検索、外部ドキュメント参照、ライブソースに対する事実確認に使用する。

### アーキテクチャ

```
ユーザー → Claude Code（司令塔・設計・統合）
  ↓
  Bash(gemini --sandbox -p "..." --output-format json 2>/dev/null) → Gemini CLI（調査員・Web 検索・事実確認）
  ↓
  Claude Code（検証・統合）
```

### 委譲優先度マトリクス（Codex Plugin と Gemini CLI の両方が有効な場合）

| タスク種別 | 委譲先 | 理由 |
|-----------|--------|------|
| 最新・外部情報の Web 検索 | **Gemini CLI** | Google 検索グラウンディング |
| 外部ドキュメント参照 | **Gemini CLI** | Web fetch + グラウンディング |
| ライブソースに対する事実確認 | **Gemini CLI** | 最新の外部エビデンスが必要 |
| コードレビュー / セキュリティスキャン | **Codex Plugin** | `/codex:review`, `/codex:adversarial-review` |
| コード生成 / リファクタリング | **Codex Plugin** | `/codex:rescue` |
| デバッグ / 根本原因分析 | **Codex Plugin** | `/codex:rescue` |
| 単一ファイル編集、設定、git、ドキュメント、QA | **Claude 直接** | シンプル、委譲不要 |

Gemini CLI のみ有効（Codex Plugin なし）の場合、Claude はコードタスクを全て直接処理し、外部調査のみ Gemini に委譲する。

**混合タスク**: リポジトリローカルの推論と外部参照が混在する場合は分割する。限定された外部サブ質問のみ Gemini に送り、ローカル推論は Claude（または Codex Plugin）で直接実行する。

### 委譲基準 — Gemini CLI

Gemini CLI に委譲するのは**以下の全条件を満たす場合のみ**:
1. Claude が持っていない**最新の外部情報**が必要（トレーニング後のデータ、ライブ URL、現在のバージョン）
2. ローカルコードベース、git 履歴、Claude のトレーニングデータからは**取得できない**
3. **限定されたリサーチサブ質問**として表現できる（オープンエンドの探索ではない）

Gemini CLI に委譲しない:
- コードレビュー、デバッグ、アーキテクチャ分析（Codex Plugin または Claude を使用）
- ローカルファイルや Claude の知識で回答可能な質問
- 広範な「X について全て調べて」というリクエスト（具体的なサブ質問に分割してから）

### コマンドパターン

全コマンドは `--sandbox` と `--output-format json` を必須とする:

| 目的 | コマンド |
|------|---------|
| Web 検索 | `gemini --sandbox -p "QUERY: <具体的な質問> OUTPUT: ソース URL 付きで回答" --output-format json 2>/dev/null` |
| 事実確認 | `gemini --sandbox -p "VERIFY: <具体的な主張> OUTPUT: True/False をソース URL 付きで" --output-format json 2>/dev/null` |
| ドキュメント参照 | `gemini --sandbox -p "LOOKUP: <API/ライブラリの具体的な質問> OUTPUT: 公式ドキュメント URL 付きで回答" --output-format json 2>/dev/null` |

### エラーハンドリング

Gemini CLI 呼び出し時、Claude は出力のエラーを確認する:
1. Bash コマンドが非ゼロで終了した場合、Gemini CLI 自体が失敗（未インストール、認証エラー、ネットワーク問題）
2. JSON 出力に `.error` フィールドが含まれる場合、API がエラーを返した（レート制限、無効なリクエスト）
3. `.response` が欠落または空の場合、クエリが有用な結果を生成しなかった

Gemini が失敗した場合（認証エラー、レート制限、パース失敗）、Claude は:
1. 失敗をユーザーに透過的に報告する（「Gemini CLI の検索に失敗: [理由]」）
2. 外部検証ができなかったことを明示する
3. 最新情報の質問（最新バージョン、現在のステータス、ライブデータ）: トレーニングデータから推測しない。未検証の回答を希望するか手動確認を希望するかユーザーに確認する
4. 鮮度が重要でない一般知識の質問: 「トレーニングデータに基づく回答、外部未検証」と明示した上で回答可能

### 検証ルール（重要）

Claude は Gemini の全出力に対して以下のルールを必ず遵守する:

1. **ソース URL 必須**: ソース URL のない Gemini 出力は未検証として扱い、事実として使用してはならない。Gemini には常にソース URL を含めるよう指示する。
2. **URL 直接検証**: Gemini の主張を最終回答に使用する前に、Claude は引用元に対して検証しなければならない:
   - `WebFetch` で引用 URL を取得し、主張がソースコンテンツに含まれていることを確認する
   - 非公式または意外な主張の場合、第二の独立したソースでも検証する
   - URL 検証が失敗または不可能な場合、その主張は省略するか「未検証の外部情報」として明示する — 裏付けられた事実として扱わない
   - 最新の外部データが必要だった主張に対して、Claude 自身のトレーニング知識だけでは検証として不十分（そもそも Gemini を呼んだ理由がそれである）
3. **パススルー禁止**: Gemini の生出力をユーザーに見せない。必ず要約・検証・統合する。
4. **スコープエビデンス**: コードベース関連の主張には `path:line` 参照、Web 主張には URL を要求する。

### 委譲ルール

1. プロンプトは自己完結的に構成する（Gemini はステートレス）
2. 各プロンプトには**限定されたスコープ**を指定する — 何を検索し、どの形式で返し、どのエビデンスを含めるか
3. API キーや秘密情報を Gemini プロンプトに含めない
4. サンドボックスモードを常に使用する（ファイル変更を防止）
