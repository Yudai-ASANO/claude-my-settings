# Best Practices

## コンテキスト管理

### CLAUDE.md サイズ制限
- 推奨: 150行以内（公式ドキュメントは200行以内を推奨）
- 超過するとアドヒアランス（指示遵守率）が低下する
- 詳細は `.claude/rules/` に分離し、CLAUDE.md はコアのみ

### コンパクション戦略
- コンテキスト使用率 50% で手動コンパクション（`/compact`）を推奨
- 自動コンパクションに頼らない（タイミングが遅すぎることがある）
- `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE` で閾値変更可能
- CLAUDE.md はコンパクション後もディスクから再読み込みされる（消えない）

### 長いセッション
- `/clear` でコンテキストをリフレッシュ
- サブタスクはコンテキストの 50% 以内で完了させる
- 大きなタスクは worktree で分離して並列実行

## コミット規律

- タスク完了時に即コミット（バッチ化しない）
- 1コミット = 1論理変更（atomic commit）
- Conventional Commits: `feat:` / `fix:` / `refactor:` / `test:` / `docs:` / `chore:`

## モデル選択ガイドライン

| モデル | 用途 | 特徴 |
|-------|------|------|
| Haiku | 軽量エージェント作業 | Sonnet の 90% の能力、3倍のコスト削減 |
| Sonnet | メイン開発 | バランス型、日常の開発作業に最適 |
| Opus | 深い推論 | 複雑なアーキテクチャ判断、デバッグ |

サブエージェントにはコスト効率の良い Haiku を検討する。

## デバッグツール

- `/doctor`: 診断ツール（設定問題のトラブルシュート）
- Playwright MCP: ブラウザテスト（トークン効率最良 ~13.7k）
- Chrome DevTools MCP: パフォーマンス分析
- スクリーンショットで視覚的問題を記録

## 並列実行

### ベストプラクティス
- 独立タスクは `isolation: worktree` で worktree 分離して並列起動する
- Sprint Contract の「依存」列で並列/逐次を判断する（依存なし → 並列、依存あり → 逐次）
- サブエージェントの起動は Agent/Task ツール経由で行う（bash 経由は動作しない）
- 各インスタンスはコンテキストの 50% 以内で作業を完了させる

### アンチパターン
| パターン | 問題 | 対策 |
|---------|------|------|
| 依存タスクの並列起動 | 競合・不整合 | Sprint Contract の依存列で逐次配置 |
| MEMORY.md の並列書き込み | 書き込み競合 | 書き込みタスクは逐次配置 |
| スコープ外ファイルの変更 | 他 generator との競合 | ファイルパス重複なき分割 |

## アンチパターン（避けるべき行動）

| パターン | 問題 | 対策 |
|---------|------|------|
| 過剰コンテキスト前方投入 | コンテキスト枯渇 | Progressive Disclosure |
| CLAUDE.md の肥大化 | アドヒアランス低下 | rules/ に分離 |
| 単純タスクに複雑なオーケストレーション | 無駄なトークン消費 | 直接実行 |
| `/memory` や `/rules` を主指示として使用 | 設計意図と異なる | CLAUDE.md で指示、memory は知見蓄積 |
| 自動コンパクション頼り | 手遅れになる | 50% で手動コンパクション |
| サブエージェント間の bash 起動 | 動作しない | Task ツール使用 |
