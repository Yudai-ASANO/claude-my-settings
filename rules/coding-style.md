# Coding Style — 推奨パターン

> このファイルは「こうすべき」推奨パターンを定義する。
> 「こうしてはならない」禁止パターンは `anti-patterns.md` を参照。

## 不変性

オブジェクトを直接変更せず、新しいオブジェクトを作成する。

- spread 演算子、`Object.assign`、`copy()`、`replace()` を使用
- 配列操作は `map`/`filter`/`reduce` を推奨（`push`/`splice` より）
- 状態管理では常に新しい状態オブジェクトを返す

## ファイルサイズ

- 推奨: 200〜400行
- 最大: 800行
- 800行を超える場合はモジュール分割を検討

## 関数サイズ

- 推奨: 20行以内
- 最大: 50行
- 50行を超える場合はサブ関数に分割

## 命名規則

- 変数・関数: その目的を正確に表す名前をつける
- 略語は一般的なもの（`id`, `url`, `config`）のみ許可
- ブール値は `is`/`has`/`can`/`should` で始める
- コレクションは複数形（`users`, `items`）

## エラーハンドリング

- エラーは必ずハンドリングする
- エラーメッセージには「何が起きたか」「何が期待されていたか」を含める
- リカバリ可能なエラーとリカバリ不可能なエラーを区別する
- 外部システム境界（API呼び出し、DB操作、ファイルI/O）では必ずエラーをキャッチ

## 入力バリデーション

- ユーザー入力は必ずバリデーションする
- システム境界（API エンドポイント、CLI 引数）で入力を検証
- 内部関数間では過剰なバリデーションを避ける（信頼境界の原則）

## 構造

- 1ファイル1責務（Single Responsibility）
- import はファイル先頭にまとめる
- export は明示的に（barrel export の濫用を避ける）
- 早期 return で深いネストを避ける
